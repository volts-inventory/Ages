## Q130 — Map-as-focus: full box, light dividers, terrain color
**Status:** Decided
**Context:** Q129 settled the section order at planet → map →
key → species → log with universal centering and the caption
moved into the planet section. The user's question shifted from
*"what's in each section?"* to *"how do I make the map the
visual focus?"* Q130 is the answer.

The constraint: terminal text rendering can't change font size,
so visual emphasis has to come from layout — whitespace,
framing, and selective color/brightness. Q130 picks five levers
and applies them all so the map sits inside an obvious frame
while everything around it recedes.

**Decision:** Six coordinated changes.

1. **Full box around the map.** Q128 added `+----` at the top
   and `|` on the left; Q130 closes the box with `|` on the
   right of every grid row and a matching `+----+` at the
   bottom. The map now reads as a fully-bounded ASCII rectangle
   — the only fully-framed artifact in the viewport.

2. **Blank lines isolating the map.** A blank row appears just
   before the `------ map ------` divider and just after the
   bottom box border. Whitespace around an artifact is the
   classic "this matters" signal; combined with the box border
   it isolates the map visually from the surrounding card text.

3. **Light dividers for non-map sections.** New `light_divider`
   helper emits a centered `· {label} ·` instead of the heavy
   `------ {label} ------`. The map keeps the heavy form so it
   visually dominates; planet / key / species / log get the
   minimal `·`-flanked label. The contrast pulls the eye toward
   the only section with full-width dashes.

4. **Default grid bumps to 32×20.** Q122's 24×16 = 384 cells
   was sized for a 30-col phone terminal; with the wider 40-col
   layout from Q125 the grid had headroom to grow. 32×20 = 640
   cells (1.67× more), runtime ~1.7× — under 4 minutes for a
   5000-year run on x86. The map gets noticeably larger
   on-screen at the same character size.

5. **ANSI dim wrap on non-map content.** When `use_color` is
   on, every non-map section (planet card, key, species, log)
   is wrapped in `\x1b[2m` … `\x1b[0m` so it renders muted
   gray on dark-mode terminals. The map keeps full brightness;
   civ markers gain `\x1b[1;…]` (bold + color) so they pop
   against everything. On `--no-color` the wrappers expand to
   empty strings and the layout is identical save the color.

6. **Terrain color codes.** New `terrain_color_code(c)` in
   `frame.rs` maps each terrain glyph to a 256-color code:
   - `~` shallow water → 39 (sky blue)
   - `≈` deep water → 27 (deep blue)
   - `▲` peak → 15 (bright white)
   - `△` low mountain → 94 (brown)
   - `▒` inland → 34 (forest green)
   - `░` coastal → 143 (sand / khaki)

   Civ overlays (centroid letter, claim digit, dispute `#`)
   keep their existing 24-hue palette and gain bold weight so
   they sit visibly above the terrain palette.

The `------ map ------` divider, the `+----+` corners, and the
`|` borders are the only heavy ASCII in the layout. Everything
else is light bullets and centered text. The eye can't *not*
land on the map.

**Why this respects the project goals.**
- *Layout-as-emphasis:* every change is structural — borders,
  whitespace, divider style. The reader's eye is guided by
  layout, not by font size we can't change.
- *Color is additive:* on a real terminal (Termius, etc.) the
  dim/bold/terrain-color combination amplifies the hierarchy.
  On a pasted-into-chat code block, the `--no-color` rendering
  still has the framed map and the light dividers — the hierarchy
  survives the loss of color.
- *Determinism preserved:* every helper is a pure function;
  the bump from 24×16 to 32×20 is config; sim state and event
  payload are unchanged.
- *Runtime budget:* +67% cells = +70% runtime — measured at
  ~21s for a 100-year run on x86, scaling linearly. The bump
  stays under common run thresholds.

**Numerical care.**
- Box top `+----+` and bottom `+----+` use
  `row_prefix_pad - 1` for left corner placement and `w` for
  the dash count, so the corners always align with the per-row
  `|` regardless of grid width.
- Right `|` per grid row is appended after the last cell,
  which means it sits at column `row_prefix_pad + w` — same
  column as the right `+` corner. Alignment preserved.
- ANSI dim is applied as a wrapper, not a per-line prefix, so
  multi-line blocks pick up dim with one escape pair instead
  of one per line.
- `terrain_color_code` returns `Option<u8>` and
  unrecognised glyphs (claim digits, blanks, `#`) fall through
  uncolored — terrain coloring is purely additive.

**Lives in:** `sim/report/src/viewport.rs` (`light_divider`
helper, blank lines around map, ANSI dim wrappers,
`-- log --` test updated to `· log ·`); `sim/report/src/frame.rs`
(top + bottom + right box borders, `terrain_color_code`,
bold civ markers); `ages/src/main.rs` (default grid → 32×20).

**Follow-ups (not blocking).**
- *Ink-saving mode.* `--no-box` flag could revert the borders
  for users on low-rez terminals where every row matters.
- *Terrain palette tuning.* The Q130 palette is opinionated;
  a `--terrain-palette warm|cool|earth` flag could let users
  pick their preference.
- *Bold map glyphs (non-civ).* Currently only civ markers get
  bold; terrain stays at normal weight (just colored). Bolding
  terrain too could over-emphasise; deferred until user
  feedback.
