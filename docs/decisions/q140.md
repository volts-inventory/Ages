## Q140 — Atomic frame paint + auto-fit grid height
**Status:** Decided
**Context:** After Q138 ostensibly fixed the viewport flicker
(home cursor only + per-line `\x1b[K` instead of full-screen
clear), the user reported the same two issues persisting on
phone Termius:

1. **Screen still flashes** every frame.
2. **Planet name divider scrolls off the top** — only the
   bottom of the planet card was visible above the map.

Root causes, identified after a closer look:

1. *Flicker is from incremental painting, not from clear.* Q138
   removed the full-screen clear, but the redraw still wrote ~42
   small `write_all` calls (one per line, plus dividers and ANSI
   wrappers). Stdout auto-flushes on every `\n`, so each line
   reaches the terminal as its own paint operation. The terminal
   redraws line-by-line — which on slow connections (mobile
   Termius) reads as a wave of flicker even without a clear.
2. *Termius's alt-screen scrolls when content exceeds height.*
   Q138's row trim (45 → 42 rows) wasn't enough — phone
   terminals with the keyboard up expose ~25-35 rows total, and
   when the cursor walks past the visible bottom Termius pushes
   the top off. The 6-row planet section disappeared first.

**Decision:** Two fixes, one per issue.

1. **Atomic frame paint via in-memory buffer.** `render()`
   builds the entire frame in a `Vec<u8>` first (one
   `write_all` per logical chunk goes into the buffer, not to
   stdout), then writes the whole buffer to stdout in a single
   `write_all` followed by one `flush`. The terminal sees ~3-4 KB
   of text arrive in one chunk and paints in a single pass —
   no incremental redraw, no flicker. `write_centered_line` and
   `write_centered_block` already accepted any `W: Write`, so
   `Vec<u8>` (which implements `io::Write`) drops in without a
   helper signature change.

2. **`run.sh` auto-fits grid height to terminal lines.** Read
   `tput lines` at script start; subtract the 22 rows of fixed
   viewport overhead (planet section + dividers + key + species
   + log) plus a 2-row margin. Clamp the result to `[8, 24]`
   and pass via `--grid-height`. Falls back to the binary's 20
   default when `tput` returns 0 (no tty, CI). Examples:
   - 50-row desktop terminal → grid_height = 24 (capped).
   - 35-row phone-keyboard-down → grid_height = 11.
   - 25-row phone-keyboard-up → grid_height = 8 (floor).

   Result: the viewport always fits the visible window, no
   matter the terminal size, and the planet section stays at
   the top of the screen.

**Why this respects the project goals.**
- *No protocol or sim-state change.* Both fixes live entirely in
  the presentation layer (`sim_report::viewport` for the buffer,
  `run.sh` for the grid sizing).
- *Determinism preserved at the sim level.* The atomic-write
  refactor changes when bytes hit stdout but not what they are.
  Grid-size variation does affect simulation determinism (a
  different grid produces a different per-cell physics state),
  so a `tput lines`-driven grid_height means a given seed +
  terminal height combination is reproducible, but the same
  seed in a different terminal height reaches a different sim
  state. That trade-off is the price of always-fits — users who
  need cross-machine determinism can pin `--grid-width 32
  --grid-height 20` directly, bypassing `run.sh`'s auto-fit.
- *Backward compatible.* The binary's defaults are unchanged
  (32×20). Only `run.sh` adapts; manual `cargo run` invocations
  still get the binary's 20-row default.

**Numerical care.**
- `Vec::with_capacity(4096)` is plenty for one frame
  (~3 KB at 32×20 with full ANSI dim/color wrappers); avoids
  re-allocation during the per-frame fill.
- `tput lines` returns the terminal's reported height; if
  `tput` is missing or returns 0 the script falls back to 20.
  Wrapped in `2>/dev/null || echo 0` so a missing tput
  never errors out the script.
- The 22-row overhead estimate is a lower bound (current
  Q138-trimmed layout). Future Q's that add rows (e.g., a
  second log section) would need to update the constant.

**Lives in:** `sim/report/src/viewport.rs::render` (Vec<u8>
buffer rewrite), `run.sh` (`tput lines` + grid_height math).

**Follow-ups (not blocking).**
- *Sim-side terminal-size detection.* The binary itself could
  read `LINES` env or query `\x1b[18t` and adapt. Then both
  `cargo run` and `run.sh` would auto-fit. Defer until a user
  asks; `run.sh` covers the common case.
- *Resize-on-the-fly.* If the terminal resizes mid-run (e.g.,
  rotating the phone), the viewport doesn't re-fit — grid size
  is fixed at run start. SIGWINCH handling is doable but adds
  complexity.
- *Reuse buffer across frames.* `Vec::with_capacity` allocates
  per-frame; could keep a buffer member on `ViewportEmitter` and
  truncate-then-reuse. Allocation cost is negligible at the
  current 50ms/6-tick cadence, so deferred.
