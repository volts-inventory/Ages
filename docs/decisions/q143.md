## Q143 — Restore frame-start screen-clear inside atomic write
**Status:** Decided
**Context:** User on phone Termius reported the planet name
divider + first card line still missing from the live viewport
even after Q138 (home-only redraw + per-line erase + frame-end
erase-below) and Q140 (atomic Vec<u8> write). The user's
screenshot also showed a duplicate column header — once before
the `-- map --` divider, once after — confirming leftover content
from the previous frame was bleeding through.

Diagnosis: `\x1b[H` alone (home cursor without clearing) doesn't
fully reset Termius's alt-screen between frames. When the new
frame is shorter than the old (e.g., the log shrinks because
recent_events has fewer items), the top of the old frame remains
visible and the new frame writes *under* it without overwriting
the rows it doesn't reach. Per-line `\x1b[K` clears tail content
on each line written, and frame-end `\x1b[J` clears below the
last new line — but neither helps if the cursor never reaches
the top rows that need clearing.

Q114 used `\x1b[H\x1b[2J` (home + full-screen clear). The
flicker that produced wasn't *because* of the clear — it was
because the screen was cleared, *then* lines were written one
by one, with stdout auto-flushing on each `\n`. The terminal
saw: clear → line 1 → line 2 → ... → line 42 as separate
paints, producing a visible wave.

Q140 fixed the per-line flushing by buffering the whole frame
into a `Vec<u8>` and writing once. Q143 piggybacks on that:
**move the screen-clear back into the buffer**. The terminal
sees `clear` + `body` arriving in one write — single paint, no
flicker. Drop the now-redundant frame-end erase-below.

**Decision:** Three small edits.

1. Replace `ANSI_HOME` (`\x1b[H` alone) with `ANSI_HOME_CLEAR`
   (`\x1b[H\x1b[2J`) emitted into the buffer at frame start.
2. Drop the `ANSI_ERASE_BELOW` write at the end of the buffer
   build (the start-of-frame clear handles it).
3. Keep the per-line `\x1b[K` (cheap safety; harmless if
   `\x1b[2J` already cleared the row).

The per-line `\x1b[K` is kept as defence against terminals that
don't honour `\x1b[2J` cleanly — it costs ~3 bytes per line and
no perceptible time, and on Termius it's a no-op after the
prior clear.

**Why this respects the project goals.**
- *Restores Q138's planet-section visibility.* The clear
  guarantees the top of the screen is empty before the new
  frame paints, so the planet-name divider + first card line
  always reach an empty row.
- *Doesn't bring back Q138's flicker.* Q140's atomic write
  delivers `clear + body` to the terminal in a single buffer
  write; the terminal renders both as one paint. No visible
  black flash.
- *Determinism preserved.* Pure presentation change; the
  NDJSON output is unchanged.
- *No regression risk for terminals that worked pre-Q143.*
  Most terminals handled the home-only redraw correctly;
  for them, adding `\x1b[2J` is a no-op (clear already-clear
  rows) plus a cheap atomic write.

**Lives in:** `sim/report/src/viewport.rs` —
`ANSI_HOME_CLEAR` const replaces `ANSI_HOME` +
`ANSI_ERASE_BELOW`; `render()` writes the clear at the start
of the buffer and drops the trailing erase.

**Follow-ups (not blocking).**
- *Diagnose what specifically broke on Termius.* Q138's home-
  only approach works on most terminals; Termius's quirk
  remains unidentified. Possibly its alt-screen retains some
  rows even after entry. Not critical now that Q143 sidesteps
  the issue.
- *Test on more mobile terminals.* a-Shell, Blink Shell,
  Prompt 3 — these all might behave differently. Q143's
  full-clear approach should work universally.
