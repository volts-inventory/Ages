## Q144 — Diff-based viewport rendering
**Status:** Decided
**Context:** User on phone Termius reported the viewport flicker
persisted through Q138 (per-line erase + frame-end erase-below),
Q140 (atomic Vec<u8> write+flush), and Q143 (full-screen clear
inside the atomic write). Each Q reduced the flicker but none
fully eliminated it. The user asked: *"should we pivot to a
known fix?"*

The known fix that real TUI applications (vim, htop, btop,
tmux, ratatui internals) use is **diff-based rendering**: build
the new frame, compare it to the previous frame line-by-line,
and emit ANSI cursor-positioned rewrites only for the rows that
*actually changed*. Unchanged rows are never touched, so the
terminal isn't asked to repaint them — flicker becomes
structurally impossible.

The viewport's frame-to-frame churn is small: the caption row
changes every frame (year/month tick), grid rows change when a
civ claims/abandons cells (occasional), the log section changes
when a notable event fires (rare). Most rows in most frames are
identical to the previous frame. A diff render saves ~95% of
the per-frame bytes and makes flicker impossible by construction.

**Decision:** Replace the atomic full-frame write with per-row
diff emission.

1. Add `prev_frame_lines: Vec<String>` to `ViewportEmitter`'s
   state.
2. `render()` builds the frame into the existing `buf: Vec<u8>`
   as before, then splits the buffer on `\n` to produce a
   `Vec<String>` of per-row content.
3. Iterate over the new lines; for each row index `i`:
   - If `prev_frame_lines.get(i) != Some(line)` → emit
     `\x1b[{i+1};1H` (CUP — cursor position) + the new line +
     `\x1b[K` (erase to end of line).
   - Otherwise → skip; the row stays as the terminal already
     drew it last frame.
4. If the previous frame had more rows than the new frame
   (e.g., the log shrank), clear the now-stale trailing rows
   with `\x1b[i;1H\x1b[K` per dropped row.
5. Single `write_all` + `flush` of the resulting cursor-move +
   content updates.
6. Save `new_lines` as `prev_frame_lines` for the next frame's
   diff.

The first frame paints fully because `prev_frame_lines` is
empty — every row counts as "changed". Subsequent frames emit
only deltas.

**Why this respects the project goals.**
- *Flicker becomes structurally impossible.* Unchanged rows
  receive zero bytes. The terminal can't flicker on something
  it isn't asked to redraw.
- *Bandwidth-friendly.* Mobile Termius over a slow network
  benefits a lot — a typical mid-run frame emits ~50-200 bytes
  instead of ~3 KB. The "atomic write" guarantee Q140 chased
  is reduced from 3 KB-atomic to 200B-atomic; far easier for
  the terminal to render in one paint pass.
- *No new dependencies.* The whole change is ~30 lines in
  `render()` and one `Vec<String>` field on the state struct.
  Uses ANSI CUP (`\x1b[{row};{col}H`) which is in every
  terminal's repertoire.
- *Same first-frame correctness.* The empty-prev-frame path
  paints every row, identical to what Q143 did.
- *Determinism preserved.* The diff is a pure function of the
  current and previous frames; same NDJSON event sequence →
  same rendered output bytes (modulo cursor-positioning order,
  which is irrelevant to a correctly-rendering terminal).

**Numerical care.**
- ANSI CUP uses 1-indexed rows; we emit `i + 1` for 0-indexed
  Rust line iteration.
- `prev_frame_lines` ownership: built fresh each frame, moved
  into self at the end. No borrow conflicts; the new frame is
  built before we touch the old.
- `String::from_utf8_lossy` on the buffer is robust: even if
  some terminal-quirk ANSI escape produces non-UTF-8 bytes
  (shouldn't happen with our codepath, but defensive), the
  diff still operates on chars and the comparison stays
  consistent across frames.
- `write_centered_line`'s trailing `\n` produces an empty
  trailing entry after `split('\n')`; we strip it so we
  don't emit a phantom row clear.

**Lives in:** `sim/report/src/viewport.rs` —
`ViewportEmitter::prev_frame_lines` field; `render()`
post-buffer logic replaced with the diff loop;
`ANSI_HOME_CLEAR` removed (no longer needed; per-row CUP
moves replace the screen-clear).

**Follow-ups (not blocking).**
- *Profile the diff cost.* `String` comparison via `PartialEq`
  is O(line_length) per row; 42 rows × ~80 chars/row = ~3.4 KB
  comparison per frame. At 50ms/tick, 12 ticks/year, this is
  noise. If the grid grows to ~64×40 we'd revisit.
- *Cell-level diff.* Currently we redraw the whole row when
  any cell in it changes. Cell-level diff (only redraw the
  delta cells within a row) would cut bandwidth further but
  the per-row approach is sufficient for our content shape.
- *Mid-frame resize handling.* If the terminal is resized
  smaller mid-run, the cursor-positioned writes might land on
  the wrong rows. Out of scope; users restart the run after
  resizing.
