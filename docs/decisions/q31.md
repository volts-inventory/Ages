## Q31 — Event ordering within a tick
**Status:** Decided (extended by Q40, Q47)
**Context:** Events are emitted from many places each tick. For byte-
for-byte determinism testing (same seed → identical event log), events
in the same tick need a canonical ordering. Q32 added hypothesis-
fitting events; Q40 added physics-engine events
(`PhysicsIntegration`) and recognition-layer events
(`PatternRecognition`) at the start of each civ-sim tick. Q47 added
`CivLifecycle` for civ founding / collapse / succession.
**Decision:** Iteration order is canonical — no write-time sort needed.
The sim walks a fixed per-civ-sim-tick phase order: TickStart →
PhysicsIntegration → PatternRecognition → CohortObservations (A) →
FigureObservations (B) → PatternDetection (C) → HypothesisTesting (D)
→ Discovery (E) → AdoptionAndDecay (F) → CapabilityEvaluation →
PopulationDynamics → CivLifecycle (M3 stub, M4 fills) → CulturalDrift
(M3 stub, M4 fills) → TickEnd. Within each phase, iterate civs by
sorted `civ_id`, then regions by sorted `region_id`, then figures by
sorted `figure_id`, then recognized-phenomenon ids by sorted id.
Combined with the existing "no `HashMap` iteration in decision paths"
rule, determinism falls out for free. The phase enum lives in
`protocol/` from M0; sub-phase ordinals can be added later without
renumbering.
**Lives in:** `docs/architecture.md` Tick → Per-tick phase order,
`protocol/` schema.
