## Q139 — Featureless-cell fallback glyphs
**Status:** Decided
**Context:** When a user ran `./run.sh` on a seed whose
per-cell terrain rolled mostly featureless (water_depth = 0
*and* elevation ≤ 0 everywhere), the live viewport rendered an
empty box — civ markers (`A`, `1`) overlaid a sea of literal
blank cells, leaving the reader unable to tell whether the cell
was on the planet at all or off the edge of the rendered area.

The cause: `render::terrain_symbol`'s fallback for the
`elev <= 0` branch returned `' '` (a single blank space). For
gas-giant compositions (`terrain_peak == 0`, no rocky surface
at all) and for low-relief rocky / sub-surface ocean planets,
nearly every cell hit that branch.

**Decision:** Replace the bare blank fallback with two
distinct glyphs that convey *what kind* of empty:

| Condition | Glyph | Read as |
|---|---|---|
| `terrain_peak == 0` | `≡` | gas-giant cloud band |
| Otherwise (`elev <= 0`, peak > 0) | `·` | featureless surface (low-relief rocky / sub-surface ocean ice / oceanic-basin floor) |

Both glyphs sit *under* the civ-overlay layer (centroid letter,
claim digit, dispute `#`), so a planet's civ activity still
reads cleanly on top of the new texture.

Q130's color palette gains entries for both:

| Glyph | 256-color code | Visual |
|---|---|---|
| `≡` (gas band) | 222 | light yellow (Saturn-ish) |
| `·` (featureless) | 244 | medium gray |

So in a color-capable terminal (Termius), a gas giant
renders as a light-yellow band texture and an Earth-low-relief
plain renders as a gray-dotted texture; civ markers stay bright
red/orange on top.

**Why this respects the project goals.**
- *Information density preserved.* The viewport's grid was
  already the dominant element; previously it lost half its
  information value when the planet had no surface terrain.
  Q139 restores that: every cell carries texture.
- *No protocol change.* `terrain_symbol`'s signature is
  unchanged; the new dispatch is internal.
- *Determinism preserved.* New glyphs are pure functions of
  existing inputs (`terrain_peak`, `elev`, `depth`).
- *Backward compatible.* The post-run report's spatial-timeline
  keyframes (which use the same renderer) gain the same
  fallback for the same kinds of worlds. No NDJSON change.

**Numerical care.**
- `terrain_peak == 0.0` exact float compare is safe: the
  planet sampler emits exactly 0 for gaseous compositions
  via integer encoding before Q32.32 conversion, so the
  comparison is bit-exact, not approximate.
- The two new color entries (222, 244) are within the 256-color
  range used elsewhere; no terminal-capability concerns beyond
  what the existing palette already requires.

**Lives in:** `sim/report/src/render.rs::terrain_symbol`
(replaces the `' '` fallback with the dispatch),
`sim/report/src/frame.rs::terrain_color_code` (adds entries
for `≡` and `·`).

**Follow-ups (not blocking).**
- *Per-row gas-band variation.* Real gas giants have
  alternating cloud bands (Jupiter's belts/zones); Q139 uses
  the same `≡` glyph everywhere on a gas-giant grid. Future
  enhancement could vary by row index for a more textured look.
- *Legend entry.* The viewport's `key` section currently doesn't
  list the new glyphs (line-width budget is tight). Most
  viewers will infer from context. Could be added if users
  ask, possibly by splitting the key into 3 lines.
- *Composition-aware overrides.* Sub-surface-ocean worlds could
  render as all-`≈` (full ocean under ice) instead of `·`, but
  that requires passing `composition` to `terrain_symbol`.
  Defer until a user notices.
