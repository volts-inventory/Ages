## Q138 — Viewport flicker, ANSI-aware centering, height trim
**Status:** Decided
**Context:** User-reported issues from running `./run.sh` on
phone Termius:

1. **Screen flashes** on every redraw. Q114-era code did
   `\x1b[H\x1b[2J` (home cursor + clear screen) before each
   frame, which produces a visible flash every time the terminal
   wipes and redraws.
2. **Row 0 was misaligned** with rows 1-19 in the grid. The
   cause: per-line centering used `chars().count()` for line
   width, but lines carrying ANSI color escapes (the colored
   civ markers `A`/`1` and the colored terrain glyphs from
   Q130) include the escape-sequence chars in that count. Row 0
   had more escape codes than terrain-only rows (because civ-
   centroid `A` adds a `\x1b[1;38;5;196m…\x1b[0m` wrapper that
   pure terrain doesn't), so the per-line pad was *smaller*
   than its neighbours' pad — row 0 shifted left.
3. **Planet section missing** at the top of the viewport. Cause:
   the full viewport was 45 rows tall (planet 6 + blank 1 +
   map 24 + blank 1 + key 3 + species 4 + log 6); a phone
   terminal with the on-screen keyboard up exposes ~25 rows;
   alt-screen mode in some terminals (Termius among them)
   auto-scrolls when the cursor walks past the bottom, so the
   top of the frame disappears.

**Decision:** Three coordinated fixes.

1. **Drop the screen-clear before each frame.** Replace
   `\x1b[H\x1b[2J` with just `\x1b[H` (home cursor only). Per
   line write is followed by `\x1b[K` (erase to end of line)
   so any leftover characters from a longer prior frame at
   the same row get cleared without a full screen wipe. End of
   frame writes `\x1b[J` (erase from cursor down) to clear
   any rows below the new frame's last line. Net effect: no
   visible flash, same artifact-free output.

2. **`visible_width` helper that strips ANSI escapes when
   counting columns.** Per-line centering switches from
   `chars().count()` to `visible_width(line)`. The function
   walks the line, treats `\x1b` as the start of an escape
   sequence, and skips chars until the next ASCII letter
   (the CSI terminator). Result: row 0 with civ-marker color
   escapes counts the same visible width as row 5 with
   terrain-only colors, and they get the same pad.

3. **Trim two rows of vertical bloat.** Default `log_lines`
   from 5 to 3 (saves 2 rows in the log section); drop the
   trailing blank line below the map's bottom border (saves 1
   row — the bottom `+----+` border + the next divider's
   `------` already separate the sections enough).

   Total: 45 → 42 rows. Doesn't fully solve the
   phone-with-keyboard issue (still over a 25-row viewport),
   but every row helps. Users can also rotate landscape, hide
   the keyboard during watching, or pass
   `--viewport-log-lines 0` for a tighter footprint.

**Why this respects the project goals.**
- *Determinism preserved.* The flicker fix changes ANSI emit
  patterns only; sim state and event log are unchanged.
- *Q126 centering still correct on text-only lines.*
  `visible_width` returns the same value as `chars().count()`
  for any line without `\x1b` — the change is only meaningful
  when escape codes are in the line.
- *Q130 framing preserved.* The map's `+----+` box top + box
  sides + box bottom + Q138's `------ key ------` divider
  immediately below still read as a clear visual frame; the
  Q130 trailing blank was redundant ornament.

**Numerical care.**
- `visible_width` is O(n) on the line; called once per line
  per frame; viewport renders at most ~45 lines per frame at
  the slowest configured cadence (12 ticks/frame). Negligible
  cost.
- The `\x1b[K` per-line erase has the side effect of clearing
  any cell-content padding past the line's content. Because
  every line is pad + content (no trailing chars), this is a
  no-op on the visible region; the terminal just resets a
  row of blank cells, which is what we want when the previous
  frame's content was longer.
- `\x1b[J` (erase below) at frame end clears any row beyond
  the cursor — important when log shrinks frame-to-frame
  (Q138's `log_lines: 3` default reduces this risk but
  doesn't eliminate it; civ catastrophes can still cause the
  log to shrink mid-run if recent_events fills then ages out).

**Lives in:** `sim/report/src/viewport.rs` — three new ANSI
const definitions (`ANSI_HOME`, `ANSI_ERASE_LINE`,
`ANSI_ERASE_BELOW`); new `visible_width` helper;
`write_centered_line` updated; `render` trim of trailing
blank; `ViewportConfig::default()` `log_lines` 5 → 3.

**Follow-ups (not blocking).**
- *Terminal-size-aware sizing.* Read `stty size` or the
  `LINES`/`COLUMNS` env vars at startup and adjust grid
  height / log lines so the viewport always fits. Currently
  the viewport is fixed-size; users adapt their terminal.
- *Compact-fit mode.* A `--viewport-compact-fit` flag could
  drop key + species sections entirely and shrink the grid
  for ultra-small terminals. Niche; defer.
- *Wide-char (CJK / emoji) width.* `visible_width` counts
  one column per non-escape `char`. If we ever ship a
  double-width glyph (CJK, some emoji), the count would
  underestimate the visible columns. Not a current concern
  since the terrain palette + bullets all stay in 1-col land.
