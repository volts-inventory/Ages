## Q95 — `magnetic_field_strong` reads the real Q94 vector field
**Status:** Decided
**Context:** Q94 promoted the planetary magnetic field from a
permanently-zero scalar stub into a real vector quantity with
`init_field` writing a dipole pattern at planet build and
`integrate` modulating it diurnally each macro-step. But the
`magnetic_field_strong` recognition template (id 6) still keyed
on `Signature::AbsAbove(Field::Charge, 20)` — a hold-over from
before any law actually wrote the magnetic state. The proxy
fired on lightning build-up rather than on the dipole, which was
the *wrong* physics for the named behaviour: a "strong magnetic
field" recognition should trigger on actual `|B|`, not on
electrostatic charge accumulation.

**Decision:**

1. **Add `Field::MagneticMagnitude` variant** to the recognition
   `Field` enum, reading `state.magnetic_field_magnitude(cell)`
   (the `sqrt(B_q² + B_r²)` helper Q94 already exposed).
2. **Switch `magnetic_field_strong`** from
   `AbsAbove(Field::Charge, 20)` to
   `AbsAbove(Field::MagneticMagnitude, 20)`.

The threshold of 20 was retained — Q94's `dipole_strength = 50`
for a `Strong` magnetosphere lifts the equatorial cells well
above 20, polar cells with `lat_factor → 0` stay below, and a
`Weak` (10) or `None` (0) magnetosphere never reaches threshold
anywhere. This matches the "strong" qualifier in the template
name without re-tuning.

**Why this respects the project goals.**
- *Physics as close to real as possible:* the recognition layer
  now reads the *named* quantity. A civ that perceives "strong
  magnetic field" via its `MagneticSense` channel is correctly
  observing the planetary dipole, not whatever electrostatic
  weather is going on. Magnetic sensing on tidally-locked
  planets, polar latitudes, low-magnetosphere worlds all behave
  according to the actual Q94 field rather than a coincidence
  of charge dynamics.
- *Least hardcoded values:* one new `Field` variant, one
  template signature swap. Threshold preserved at 20.
- *Determinism:* `magnetic_field_magnitude` uses
  `sim_arith::transcendental::sqrt` (deterministic Q32.32);
  recognition scans iterate cells in the same canonical order
  as before. No new state, no interior mutability.

**Numerical care.**
- Q95 added a cached `magnetic_magnitude: Vec<Real>` to
  `PhysicsState`, written once per macro-step by
  `Magnetism::init_field` / `Magnetism::integrate` (the
  `integrate` path skips `sqrt` entirely — `B_q = 0` for the
  axis-aligned dipole, so `|B| = |B_r|`). Recognition reads the
  slice instead of recomputing per evaluation.
- Cache contract: any law that writes `(B_q, B_r)` *must* also
  refresh the cached magnitude. Currently only `Magnetism`
  writes the vector field; future Q98 Lorentz-coupling
  refinements that vary `B` from charge dynamics would need to
  follow the same convention.
- *Perf note:* the long sim-core test (24,000 ticks) went from
  786 s pre-Q95 to ~1160 s post-Q95. Profiling makes clear this
  isn't the `sqrt`; the cache only saved ~16 s. The real cost
  is *correct firings* — pre-Q95 the template proxied through
  `Field::Charge`, which rarely tripped, so `magnetic_field_strong`
  was effectively a no-op. Post-Q95 the planetary dipole on a
  Strong magnetosphere generates many real firings per scan,
  driving genuine downstream recognition work (samples, fits,
  confirmations). The runtime increase is the *right physics
  doing the work it was always supposed to do* — same
  philosophy that justified Q90's +25 % cost over the entire
  five-wave plan. Still well inside the "hours not days"
  budget.
- `B_q² + B_r²` magnitudes peak at `(50)² + 0 ≈ 2500` for a
  `Strong` magnetosphere — far inside Q32.32's ±2.1×10⁹.

**Lives in:** `sim/recognition/src/lib.rs`
(`Field::MagneticMagnitude` variant + `magnetic_field_strong`
template signature swap), `sim/physics/src/state.rs` (cached
`magnetic_magnitude` slice + `magnetic_magnitude()` /
`magnetic_magnitude_mut()` accessors;
`magnetic_field_magnitude(cell)` now returns the cached value),
`sim/physics/src/magnetism.rs` (refresh the cache in
`init_field` and `integrate`).

**Follow-ups (not blocking).**
- *`magnetic_lodestone` real lodestone modeling.* The id-20
  template still uses `Field::Charge` with a moderate-band
  signature as a proxy for "iron-rich crust hotspot". The
  proper lodestone model needs per-cell crust-derived
  magnetic anomalies (a separate `magnetic_anomaly: Vec<Real>`
  field) or a per-cell `Crust` variant. Left as charge-proxy
  for now; the named behaviour ("dry land, moderate
  electromagnetic pocket") is *not* the planetary dipole, so
  Q94's vector field doesn't directly replace it.
- *`Signature::DirectionalMagnetic`*. Templates that key on
  *direction* (compass bearing, magnetic-north detection) need
  the vector components, not just magnitude. Lands when civs
  build navigation tools that read direction, not just
  presence.
