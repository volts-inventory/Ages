## Q33 — Functional form vocabulary
**Status:** Decided
**Context:** Q32 lets civs propose functional forms but only from an
authored vocabulary. Which forms are in M3's set?
**Decision:** 12 hardcoded fixed-arity single-variable forms, with the
architectural shape of a *tiered vocabulary* (each form is gated by an
`available_forms(figure) -> Set<Form>` query at hypothesis-proposal
sites). For M3 the gate is a placeholder that returns all 12 at T0;
real sensorium / tech gating lands in Q53.

**Forms (12):** `constant`, `linear`, `periodic_sine`, `inverse_square`,
`exp_decay`, `exp_growth`, `logistic`, `polynomial_2`, `polynomial_3`,
`threshold_step`, `power_law`, `logarithmic`.

**Constraints:**
- Fixed arity per template (e.g. `linear` is `a*x + b`, two params;
  `polynomial_3` is four). No variable-arity templates.
- Single-variable inputs only in M3. Multi-variable regression is a
  separate design pass deferred past M3.
- Hardcoded in `sim/civ`. Move to `data/forms/*.toml` only if the list
  starts churning.

**Why tiered shape now, even with placeholder gating:** every
hypothesis-proposal site needs to call the same `available_forms(...)`
query. Building that seam now is much cheaper than retrofitting
gating across every fit site later. The unlock *table* is what we're
deferring (Q53), not the architectural seam.

**Follow-ups:**
- **Q53** — Functional-form unlock table (sensorium / tech-tier
  driven). Lands once M3's sensorium-extending tech design is
  concrete enough to attach unlocks to.
- Authorable additions to the 12-form list deferred indefinitely.

**Lives in:** `sim/civ` README (hypothesis pipeline) when implemented
in M3, plus the unlock table in Q53 when that lands.
