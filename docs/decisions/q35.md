## Q35 — Refinement triggers and rules
**Status:** Decided
**Context:** A confirmed relation can have its form revisited as data
accumulates. When? With what cooldown? Can a figure propose only one
new form, or several until one fits?
**Decision:** Tolerance-based trigger restated in confidence terms,
Occam-adjusted candidate selection, explicit probation lifecycle.

- **Confirmation threshold (anchors Q34):** A relation is confirmed
  when `confidence ≥ exp(-1) ≈ 0.368` (equivalently
  `residual ≤ tolerance`). This was implicit in Q34; recorded here
  because Q35 references it.
- **Refinement trigger:** `confidence ≤ exp(-2) ≈ 0.135`
  (equivalently `residual ≥ 2 × tolerance`) sustained for 50 ticks
  on the confirmed form.
- **Cooldown:** 100 ticks per `(figure, relation)` between refinement
  attempts. Distinct relations on the same figure are independent.
- **Concurrency:** at most one pending refinement per
  `(figure, relation)`. A figure may have multiple distinct relations
  in refinement simultaneously.
- **Candidate selection:** re-fit every form in
  `available_forms(figure)` (per Q33) on the current observation
  window. Score each candidate
  `score = confidence − λ × param_count` with `λ = 0.02` as the
  starting penalty (poor man's BIC; prevents drift toward highest-
  arity forms). Propose only if
  `score_best > score_current + 0.05`. Otherwise no proposal — the
  cooldown still resets to avoid retry-storms.
- **Probation lifecycle:**
  - On trigger: emit `relation.refinement_proposed{figure, relation,
    old_form, new_form, sample_count}`. The old form remains the
    civ-visible truth during probation.
  - During probation: accumulate `k_form` additional samples
    (per-form floor from Q34), refit the proposed form on the full
    data window.
  - Confirm: `confidence ≥ exp(-1)` ⟹ emit
    `relation.refinement_confirmed{...}`, switch the relation to the
    new form.
  - Reject: 200-tick probation window expires without confirmation
    ⟹ emit `relation.refinement_rejected{reason}`, revert,
    start cooldown.
- **Events:** `relation.refinement_proposed`, `.confirmed`,
  `.rejected` with attribution fields (figure id, relation id, old
  form, new form, residual, sample count). Required for the post-run
  report's "wrong then right" arcs.

**Tuning constants** (`λ`, switch margin 0.05, sustained-trigger 50
ticks, cooldown 100 ticks, probation 200 ticks): pinned at these
defaults for M3 reference runs; calibration against accumulated runs
falls under Q28's M3-tuning umbrella.

**Why not full BIC (Alt B):** principled but requires speculative
candidate fits every tick to detect when ΔBIC crosses 6. The Occam-
adjusted score above is BIC-flavoured without per-tick speculation —
candidate fits run only on trigger. Full BIC migration is reasonable
once the pipeline is empirically warm.

**Lives in:** `sim/civ` refinement module when implemented in M3.
Event types defined in `protocol/`.
