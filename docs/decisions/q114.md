## Q114 — Viewport planet card + scrolling event log
**Status:** Decided
**Context:** Q62 shipped the live ASCII viewport: a 12×8 grid
showing per-civ territory on the planet's terrain, refreshed
every N ticks. It's intentionally minimal — one layer (territory)
at one cadence — and the rich event data (foundings, collapses,
catastrophes, knowledge transmissions) lives in the NDJSON log
or the post-run report. For mobile / portrait-terminal users
that's a wasted opportunity: the screen has 30+ rows available
and the viewport only uses ~12 of them. The rest could surface
*what's happening right now* without leaving the live display.

**Decision:** Two additive features on `ViewportEmitter`:

1. **Planet card** (2 lines above the caption) showing the
   static planet identity:
   ```
   aqueous · reducing atm · none magnetosphere · 4 moons
   350K mean ±25K · 32h day · tilt 45° · year 13mo
   ```
   Static for the run since `Planet` is emitted once. Lets a
   viewer at a glance see "what kind of planet is this" without
   leaving the live display.

2. **Scrolling log tail** (N lines below the frame) that
   captures *significant* events as they fire:
   ```
   recent events:
   [year   142] civ 1 founded (10 cells)
   [year   208] catastrophe: volcanic on civ 1
   [year   215] civ 1 → civ 2 transmitted relation 17
   [year   281] civ 3 unlocked celestial_arc_thermal
   [year   312] civ 1 collapsed (food_crisis)
   ```
   Filtered to `CivFounded`, `CivCollapsed`, `CatastropheFired`,
   `KnowledgeTransmitted`, `ConflictResolved`, `TechUnlocked`.
   Per-tick noise (Tick events, recognition firings, hypothesis
   churn) is excluded — those would scroll too fast to read on
   a phone screen.

Both default-on with sensible defaults (`show_planet_card =
true`, `log_lines = 5`). Disable with `--no-planet-card` and
`--viewport-log-lines 0` for cramped terminals.

Padding rows fill out the log section when fewer events have
fired than `log_lines`, so the visible region stays a stable
height across frames (avoids the planet jumping up the screen
as the log fills).

**Why this respects the project goals.**
- *Mobile-friendly density:* a portrait-phone terminal (40×30)
  now uses ~21 rows for planet card + frame + log instead of
  ~12 for frame alone. Same screen, more information.
- *Read-only consumer of the event stream:* the log tail
  filter is a pure function of the event variant. No state
  mutation that affects sim determinism. The `recent_events`
  deque lives entirely in the viewport process.
- *Same `Emitter` pattern:* changes are confined to
  `ViewportEmitter::apply_state` (push to deque) and
  `::render` (write planet card + log lines). No new traits,
  no new fields on `Event`.
- *Determinism preserved:* given the same NDJSON event stream,
  the same log tail and planet card render. Q36 stays intact.
- *Existing CLI compat:* the new flags are opt-out; users
  running `--cli=viewport` see the richer display without
  changing their command.

**Numerical care.**
- `recent_events` is a `VecDeque<String>` capped at
  `cfg.log_lines`. O(1) push/pop. ~64 bytes per entry × 5 = 320
  bytes — negligible.
- Planet card formatting uses the existing `q32_to_f64` helper
  for display values. No new conversion paths.

**Lives in:** `sim/report/src/viewport.rs` (new
`show_planet_card` / `log_lines` config fields, new
`recent_events` deque on the emitter, new `log_message` filter
+ `planet_card` formatter, render-path additions, 2 new tests),
`ages/src/main.rs` (new `--no-planet-card` and
`--viewport-log-lines` CLI flags + plumbing).

**Follow-ups (not blocking).**
- *Layer cycling.* Currently the frame shows civ territory only.
  Pressing a key (or `--viewport-layer=temperature`) could swap
  the cell-color mapping to temperature, vapour density, wind
  magnitude, magnetic magnitude. Same grid, different data.
- *Color-coded log levels.* Founding = green, collapse = red,
  catastrophe = yellow. ANSI escapes alongside the existing
  256-color cell palette.
- *Configurable filter.* `--viewport-log-include=
  founded,collapsed,tech` to let the user pick what counts as
  significant.
- *Wrap detection.* On terminals narrower than the planet
  card's ~60 chars, the second line wraps. A `tput cols` check
  could collapse to a single denser line on narrow screens.
