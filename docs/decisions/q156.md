## Q156 — Per-terrain habitability multipliers + claim-eligibility gate
**Status:** Decided
**Context:** Q147 + Q154 added per-cell terrain glyphs (`≈` deep
ocean, `~` shallow sea, `░` coast, `▒` inland, `·` plain, `△` hill,
`▲` peak, `≡` gas band) that the renderer in `sim/report` paints
into the live viewport map. The `sim/civ` lifecycle, however,
treated every cell as undifferentiated land — `Civ::carrying_capacity`
summed `Substance::Fuel` across `claimed_cells` with no knowledge
of whether a cell was deep ocean, mountain peak, or fertile plain,
and `compute_territory` BFS'd outward from `figures[0].cell_assignment`
without checking the terrain underneath. On the default 36×30
grid the figure-derived centroid is a small integer (frequently
cell 0); seed 1 puts cell 0 inside the deep-ocean band, so the
inaugural civilisation literally founded its capital on the
sea floor and claimed nine more deep-ocean cells alongside it.
User feedback: "shouldn't terrain type have implications for
habitable for species?" — yes.

**Decision:** Two coupled changes wired through `sim/world` /
`sim/civ` / `sim/core`.

1. **Per-glyph habitability multiplier.** A new
   `sim_world::habitability_multiplier(glyph: char) -> Real`
   table (Q32.32 fixed-point throughout — no `f64` enters the
   sim path):

   | glyph | terrain      | multiplier | rationale                     |
   |-------|--------------|-----------:|-------------------------------|
   | `≈`   | deep ocean   |       0.00 | uninhabitable                 |
   | `~`   | shallow sea  |       0.05 | fishing margin only           |
   | `░`   | coast        |       1.20 | rich (fishing + farming)      |
   | `·`   | plain        |       1.00 | baseline                      |
   | `▒`   | inland       |       0.90 | generic                       |
   | `△`   | hill / low mtn |     0.60 | rough but workable            |
   | `▲`   | peak         |       0.10 | nearly uninhabitable          |
   | `≡`   | gas band     |       0.00 | uninhabitable                 |

   Plus `sim_world::terrain_glyph_at(state, planet, cell) -> char`
   (mirrors the renderer's `sim/report/src/render.rs::terrain_symbol`
   classification — depth > 100 m = `≈`, any depth = `~`,
   relative-elevation cuts at 0.7 / 0.4 of `terrain_peak`,
   coast-vs-inland by checking the four cardinal axial neighbours
   for water) and a `cell_habitability` convenience wrapper.
   `Civ::carrying_capacity_with_terrain`,
   `Civ::seasonal_carrying_capacity`, and `Civ::cell_capacity`
   multiply each per-cell fuel contribution by the cell's
   habitability multiplier; `Civ::check_collapse_with_terrain`
   uses the same gated capacity for food-security so a civ
   stranded on peaks / shallow sea starves on the schedule its
   habitable claim deserves rather than the raw fuel column's
   optimistic story. Q65 inter-cell migration inherits the
   gated `cell_capacity` automatically, so high-pressure cells
   shed pop toward better terrain without any explicit migration
   rule for "leave the mountain".

2. **Claim-eligibility gate.** A new
   `sim_world::is_claimable_multiplier(mult) -> bool` returns
   `true` iff `mult ≥ 0.05`. Deep ocean (`≈`, `0.00`) and gas
   band (`≡`, `0.00`) fall below the threshold; shallow sea
   (`~`, `0.05`) and everything richer pass it. The
   `compute_territory` BFS in `sim/core` skips cells that fail
   the gate (they're treated as walls — never claimed and never
   expanded through, so a deep-ocean band acts as a barrier
   between continents). Capital cells must pass the gate too:
   a new `pick_habitable_cell(seed, grid, state, planet)` walks
   outward from the figure-derived seed via canonical-order BFS
   until it finds a claimable cell, then sim/core overwrites
   `civ.territory_centroid` with the result. All three founding
   sites (inaugural civ, Q47 stateless-refound, Q48-v3
   breakaway) call `pick_habitable_cell` immediately before
   `compute_territory` so a successor whose figure-derived
   centroid lands in deep ocean still founds on real land.

**Threshold choice (`0.05`).** Picked so deep ocean and gas band
— the only `0.00` multipliers — fail the gate while shallow sea
(`~`, `0.05`) is accepted as marginally claimable. Peaks (`▲`,
`0.10`) stay claimable but contribute almost nothing to capacity
— a civ that BFS-expands through a high-peak band picks up the
cells but gets ~10% capacity per cell from them, which reads
narratively as "the empire's writ runs to the mountains, but
nobody actually lives up there".

**Why this respects the project goals.**
- *Determinism.* The multiplier table is `const Real` values,
  the BFS gate reads from the same elevation / water_depth
  fields `init_planet` writes once at planet seed, and the
  `pick_habitable_cell` walk uses the same canonical-order hex
  BFS the existing `compute_territory` uses (sorting neighbours
  by cell id at each step). Same seed → same pick → same claim
  set, byte-for-byte.
- *Q32.32 / Real arithmetic.* All multipliers are
  `Real::from_ratio(num, denom)` with small integer numerators
  / denominators. No `f64` enters the sim path; the renderer's
  `f64` classification in `sim/report` is reproduced inside
  `sim/world` against the same elevation values via Q32.32
  comparisons (`elev > peak_70 = peak * 7 / 10`).
- *Renderer alignment.* `terrain_glyph_at` reproduces
  `sim/report/src/render.rs::terrain_symbol`'s classification
  exactly, so the habitability decision sim/civ makes for cell
  C agrees with the glyph the user sees on the viewport map.
  No "ghost terrain" where the sim treats a cell as ocean but
  the viewport draws land or vice versa.
- *Backward compatibility.* `Civ::carrying_capacity` (planet-
  less) and `Civ::check_collapse` (planet-less) stay in place
  for unit tests that don't thread a `Planet` through. The
  production hot path in `sim/core` calls the
  `*_with_terrain` siblings; only the per-tick civ-lifecycle
  phase needed the swap. Pre-existing tests pass byte-identical.
- *Catastrophe / Q65 migration.* The catastrophe `disease_fires`
  trigger still reads `carrying_capacity` (planet-less) — the
  habitability multiplier doesn't change *which* civs trigger
  disease, only *where* their pop concentrates. Q65 migration
  uses `cell_capacity`, which now applies habitability, so
  high-pressure cells naturally shed toward fertile cells.

**Smoke test.** Seed 1 / 36×30 / `--years 5`: cell 0 (the
figure-derived founding seed) has elev = 1608 m, depth = 714 m
→ glyph `≈` (deep ocean). Pre-Q156 the inaugural civ claimed
ten cells starting at cell 0; post-Q156 the centroid relocates
to cell 33 (inland, glyph `▒`) and all ten claimed cells are
inland (`▒`) or hill (`△`) — none deep ocean / gas band.

**Numerical care.**
- Threshold `0.05` is encoded as
  `pub const Q156_CLAIM_THRESHOLD_NUM: i64 = 5; … _DEN: i64 = 100`
  so a future tuning pass can find the constants by name.
- `pick_habitable_cell` falls back to the seed cell if the
  entire grid is uninhabitable (gas-giant / open-ocean planets).
  The civ-founding pipeline still emits `CivFounded`; the civ
  then collapses on `food_crisis` or `TerritoryTooSmall` per
  Q49 / Q155 as before, rather than panicking.
- The BFS gate in `compute_territory` always includes the
  centroid itself (even if uninhabitable, as a safety valve)
  so a misplaced capital doesn't silently produce an empty
  claim set. In production `pick_habitable_cell` runs first so
  this branch is a belt-and-braces guard for synthetic test
  inputs.

**Lives in:**
- `sim/world/src/lib.rs` — new `habitability_multiplier`,
  `is_claimable_multiplier`, `terrain_glyph_at`,
  `cell_habitability` public functions; `Q156_CLAIM_THRESHOLD_*`
  constants.
- `sim/civ/src/lib.rs` — new `Civ::carrying_capacity_with_terrain`
  and `Civ::check_collapse_with_terrain` methods;
  `seasonal_carrying_capacity` and `cell_capacity` updated to
  multiply per-cell contributions by the cell's habitability
  multiplier; tests for the multiplier table, the claim
  threshold, and an end-to-end smoke (seed 1 / 36×30 → no
  claimed cell is `≈` / `≡`, plain vs terrain-aware capacity
  differ).
- `sim/core/src/lib.rs` — `compute_territory` extended with
  `state` + `planet` + habitability gate; new `pick_habitable_cell`
  helper; inaugural / stateless-refound / breakaway founding
  paths all call `pick_habitable_cell` before `compute_territory`;
  the per-tick collapse check uses `check_collapse_with_terrain`.
- `docs/decisions/q156.md` — this file.
- `docs/decisions/INDEX.md` — Q156 added to "Latest decisions"
  and to the sim-core decided section.

**Follow-ups (not blocking).**
- *Tune multipliers.* The published table is a first pass; a
  reference-run calibration may find e.g. shallow sea at 0.05
  too generous (a civ that BFS-expands across a `~` band picks
  up real capacity) or peaks at 0.10 too generous (mountain
  empires shouldn't grow). Tracked under Q28's tuning umbrella.
- *Per-substrate habitability.* Aqueous-substrate civs would
  rationally treat `~`/`░` differently from
  hydrocarbon-substrate civs (whose "ocean" is liquid methane).
  The current single table treats `≈`/`~`/`░` as
  Earth-water-and-coast for every substrate; a Q156-v2 could
  read `Planet::biosphere` / `Composition` and re-key the
  table per-substrate. Out of scope for the static-geometry
  pass.
- *Climate-coupled habitability.* Q15 (climate) and Q67
  (seasonal physics) already adjust per-cell capacity; Q156
  is *static* on the terrain glyph. A future pass could
  layer a temperature-derived habitability factor on top so
  e.g. a coastal cell at -50 K reads as worse than a coastal
  cell at +20 K. Out of scope per the brief.
- *Disease trigger.* `catastrophe::disease_fires` still uses
  the planet-less aggregate `carrying_capacity` for the 80%
  crowding check. Could be threaded through to use the
  terrain-aware capacity for tighter alignment with where
  population actually concentrates. Trivial swap; deferred to
  avoid touching the catastrophe path in this Q.
