## Q146 — Centroid letter wins disputes; revert Q144 diff render
**Status:** Decided
**Context:** User screenshot from Lumen-h (seed 495, year 56)
showed two real bugs in the live viewport:

1. **Capital letter missing at the contested centroid.** Q142
   surfaced `#` at any cell claimed by 2+ civs, even if it was a
   centroid for one of them. The user pushed back: they wanted
   to *still see a capital marker* at a contested capital,
   accepting that one of the two civs' letters wins.

2. **Stale rows from previous frames bleeding through.** With
   Q144's diff render, the screenshot showed two captions
   simultaneously — `Y56 M6 · 2 civ · 2F/0C` (current) and
   `Y17 M0 · 1 civ · 1F/0C` (from a frame 39 sim-years earlier)
   — alongside duplicate map rows. Diff render with absolute
   `\x1b[N;1H` cursor positioning depends on the terminal
   honouring those positions consistently in the alt-screen;
   Termius (and likely some other mobile terminals) didn't, and
   accumulated stale content alongside live content over many
   frames.

**Decision:** Two coordinated reverts, plus a small ordering tweak
on the centroid map.

1. **Centroid wins over dispute at the same cell.** In
   `frame.rs::render_world_frame_inner`, the centroid lookup
   runs before the multi-owner dispute check — Q142's order is
   reversed back to the pre-Q142 dispatch. Surrounding
   non-centroid contested cells still render as `#`, so the
   dispute *is* visible along the borders without erasing the
   capital.

2. **Older civ wins shared centroids.** Pre-Q142 used
   `centroids.insert(...)`, which always overwrote with the
   later-iterated civ's id. With BTreeMap's ascending civ_id
   iteration, civ 2 always overwrote civ 1 — `B` rendered, `A`
   was silently masked. Q146 switches to
   `centroids.entry(...).or_insert(civ.civ_id)`: civ 1 inserts
   first and wins, civ 2's redundant insert is ignored. So at
   the contested capital, civ 1's `A` persists. The Q47-era
   model is "civ 2 emerged within civ 1's old territory" — keeping
   the predecessor's marker visible as the heir overlaps reads
   correctly.

3. **Revert Q144's diff render.** Drop `prev_frame_lines`,
   the per-row CUP loop, and the new-vs-old comparison. Restore
   Q143's atomic clear-and-paint: each frame begins with
   `\x1b[H\x1b[2J` (home + full-screen clear) prepended to the
   Q140 atomic `Vec<u8>` write, so the terminal sees clear and
   content as one chunk and paints both as a single pass.

   Q144's diff render was elegant on terminals that consistently
   honour `\x1b[N;1H` in alt-screen mode — but on Termius, where
   absolute cursor positioning could land in different visible
   regions across frames, stale content from earlier frames
   stayed on screen. The atomic clear-and-paint is less efficient
   (~3 KB/frame vs the diff's ~200 B), but it's reliable across
   every alt-screen-supporting terminal we've tested. The user's
   pacing (50 ms/tick × 6-tick frames = 300 ms/frame) means a
   3 KB write at 10 KB/s mobile network finishes in ~300 ms —
   still well within the frame budget.

**Why this respects the project goals.**
- *User-controlled aesthetics.* The user explicitly asked for
  the capital letter to remain visible during conflicts. Q146
  delivers that without removing the dispute signal entirely.
- *Predecessor identity preserved.* Showing the older civ's
  letter at the contested capital matches the Q47 succession
  model — civ 2 overlaps civ 1's territory; civ 1's marker
  stays put until civ 1 actually collapses, at which point
  civ 2's `B` would surface (since civ 1 is no longer in
  `frame.civs` to populate the centroid map).
- *Reliability over elegance.* The diff render was a clean idea
  but assumed terminal behaviour we couldn't verify on the
  user's actual device. Atomic clear-and-paint is the
  long-known TUI fallback that just works.
- *Determinism preserved.* Same NDJSON event sequence → same
  rendered output bytes (modulo cursor state, which doesn't
  affect what the terminal displays).

**Numerical care.**
- `centroids.entry(...).or_insert()` is O(log n) on civ count
  (≤ ~20 in practice); negligible.
- The `framed: Vec<u8>` allocation is `buf.len() + 8` bytes —
  just enough for the `\x1b[H\x1b[2J` prefix without realloc.

**Lives in:**
- `sim/report/src/frame.rs` — centroid `or_insert` builder;
  centroid lookup before dispute check in render dispatch.
- `sim/report/src/viewport.rs` — `prev_frame_lines` field
  removed; diff loop replaced with atomic clear+content write;
  `ANSI_HOME_CLEAR` const restored.

**Follow-ups (not blocking).**
- *Diff render with periodic full-repaint.* If we ever revisit
  the diff approach, mixing it with a full-clear every N frames
  (or on civ_count change) would self-correct against any
  accumulated terminal state drift. Defer until performance
  pressure demands it.
- *Per-civ centroid colour at shared cells.* When the older
  civ's letter wins, the newer civ's centroid is invisible.
  Could mark the cell with a different background colour or
  underline to hint that more than one civ centres there.
  Pure ornament; defer.
