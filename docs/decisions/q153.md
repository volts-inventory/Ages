## Q153 — Viewport polish: redundant key entries, conflict log dedup, bigger default grid
**Status:** Decided
**Context:** Three small viewport rough edges accumulated after
Q150 (per-civ sidebar panels) shipped:

1. The global `--- key ---` block still lists `A=cap · 1=civ`,
   but every per-civ panel in the sidebar now carries its own
   `{letter}=cap · {digit}=civ` identity line — the global entry
   is duplicated information.
2. Q148 filtered out `ConflictResolved` events with
   `loser_defeated=false` (mid-war skirmishes), but the log still
   floods on multi-cell wars: Q66 emits one
   `loser_defeated=true` event *per cell* the loser drops below
   the defeat floor. The user reported seeing
   `y81 conflict resolved (civ 2 vs civ 1 — civ 1 defeated)` and
   `y87 conflict resolved (civ 2 vs civ 1 — civ 1 defeated)` on
   the same war.
3. The default grid (32×20 since Q130) leaves vertical headroom
   on phones held in portrait — the user has space for more
   cells without the planet section scrolling off.

**Decision:** Three independent edits, all in
`sim/report/src/viewport.rs` plus the CLI defaults in
`ages/src/main.rs`.

**1. Drop `A=cap · 1=civ` from the global key.** The global key
keeps only glyphs that aren't surfaced per-civ:

```
#=war · ~sea · ≈deep
▲peak · △hill · ▒land
░coast · ·=plain · ≡=gas
```

`#=war` stays — it's the disputed-cell glyph, not tied to any one
civ — so it remains a global concept. The key drops from 4 lines
to 3, recovering one row of sidebar height for per-civ panels.

**2. Dedup `ConflictResolved` log lines per (winner, loser)
pair.** New field on `ViewportEmitter`:

```rust
wars_logged: BTreeSet<(u32, u32)>,
```

The `Event::ConflictResolved` arm of `log_message` now:
- early-returns `None` when `loser_defeated == false` (Q148 rule
  preserved);
- early-returns `None` when the `(winner_civ_id, loser_civ_id)`
  pair already sits in `wars_logged` (Q153 dedup);
- inserts the pair into the set on the first emit.

`Event::CivCollapsed` clears every pair touching the collapsing
civ_id from the set (`retain(|(w, l)| *w != id && *l != id)`) so
a re-emerged civ_id (Q48 succession) can re-trigger a fresh
"conflict resolved" line.

To mutate `wars_logged` from the log path, `log_message` was
refactored from an associated `fn` to a `&mut self` method. The
`prev_cosmology: &BTreeMap` parameter (Q148) was dropped — the
method now reads `self.civ_cosmology` directly. Call site in
`apply_state` collapses from `Self::log_message(ev,
&self.civ_cosmology)` to `self.log_message(ev)`.

**3. Bump default grid 32×20 → 36×30.** The user-facing default
in `ages::main::parse_args`:
- `grid_width: Option<u32> = Some(36)` (was 32),
- `grid_height: Option<u32> = Some(30)` (was 20).

Cell budget: 32×20 = 640 → 36×30 = 1080 (~1.7× the cells, ~1.7×
per-tick cost; a 5000-year run stays under ~7 minutes on
commodity x86).

`MAP_WIDTH` follows: 32-cell rows render as `{r:>2}|` (3 prefix)
+ 32 cells + `|` = 36 cols → 36-cell rows = 40 cols. So
`MAP_WIDTH = 40` (was 36). Total viewport: `MAP_WIDTH + 3 (gap)
+ SIDEBAR_WIDTH` = 40 + 3 + 30 = 73, rounded to `VIEWPORT_WIDTH
= 74` (was 70). Still fits Termius portrait phone column count
(≥ 80 cols typical).

`RunConfig::dev` (`sim/core/src/lib.rs`) keeps its 12×8 default —
that path is exclusively for sim-core tests where the small grid
keeps test runtime trivial; the CLI always overrides it via
`parse_args`'s explicit defaults.

`--help` text refreshed: `default 32 / default 20` → `default 36
/ default 30`; "fits the 32×20 grid in ~36 cols" → "fits the 36×30
grid in ~40 cols".

**Why this respects the project goals.**
- *Map is the focus.* The bigger grid lets terrain features
  resolve with more cells per kilometre, surfacing climate bands
  and mountain ranges that 32×20 collapsed.
- *Sidebar stays glanceable.* Dropping the redundant `A=cap ·
  1=civ` line from the global key recovers one row, partially
  offsetting the per-civ panels expanded by 10 extra map rows.
- *Log is signal, not noise.* The dedup latch reduces a 5–8-line
  multi-cell-war flood to a single line per outcome, matching
  the "narrative beats" framing the log was designed for.
- *Determinism preserved.* `wars_logged` is keyed on civ_ids
  (deterministic from `CivFounded`) and updated in event order;
  same NDJSON in → same log lines out.

**Numerical care.**
- `wars_logged.insert` / `wars_logged.retain` on `BTreeSet`:
  O(log n) per op, n ≤ a few dozen pairs.
- `36 × 30 = 1080` cells fits in `u32::MAX` cell-id space with
  enormous margin; no overflow concerns.

**Lives in:**
- `sim/report/src/viewport.rs` — `wars_logged: BTreeSet<(u32,
  u32)>` field on `ViewportEmitter`; `log_message` refactored to
  `&mut self`; `MAP_WIDTH` 36 → 40, `VIEWPORT_WIDTH` 70 → 74;
  `build_sidebar_lines` legend trimmed from 4 lines to 3 with the
  `A=cap · 1=civ` entry removed.
- `ages/src/main.rs` — `grid_width` / `grid_height` defaults
  bumped 32 → 36 and 20 → 30; `--help` text refreshed to match.

**Smoke test (seed 495, Lumen-h, 100 years):**
- Map renders 36×30 (rows `0..29`, cols `0..35`).
- Key shows the trimmed 3-line form.
- Underlying NDJSON has 8 `loser_defeated=true` events on the
  civ 2 vs civ 1 war; viewport log shows exactly one
  `y50 conflict resolved (civ 2 vs civ 1 — civ 1 defeated)` line.

**Follow-ups (not blocking).**
- *War-end timestamp.* The dedup latch fires on the *first*
  cell-flip-with-defeat, which is when the loser crosses the
  defeat floor — close enough to "the war ended" for narrative
  purposes, but a future pass could surface
  `loser_defeated=true` only once Q66 emits a dedicated
  `WarEnded` event. Defer until that event exists.
- *Bump `RunConfig::dev` to 36×30.* Currently still 12×8 for
  test-suite speed. Could split into `dev` (small) and
  `default` (CLI-matching) constructors. Defer — the CLI sets
  explicit values and tests prefer the small grid.
