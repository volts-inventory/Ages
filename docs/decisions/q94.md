## Q94 — Planetary magnetic field as a vector quantity
**Status:** Decided
**Context:** Pre-Q94 the magnetic field was a scalar magnitude
(`magnetic_field: Vec<Real>`) with no direction. The recognition
library's `magnetic_field_strong` template even keyed on
`Field::Charge` as a proxy because nothing actually wrote the
magnetic state — the field was permanently zero, the accessor
existed but was unused.

Real planetary magnetic fields have direction. They're
approximately *dipole* patterns aligned with the rotational
axis: the horizontal surface component is strongest at the
equator (pointing from the south magnetic pole toward the
north), tapering to zero at the poles where the field becomes
purely vertical. Without direction we can't represent compass
behaviour, particle deflection, or aurora geometry.

**Decision:** Two pieces:

1. **Vector promotion in `PhysicsState`.** Replace the scalar
   `magnetic_field: Vec<Real>` with two slices `magnetic_b_q`
   and `magnetic_b_r` (axial-coordinate components, same basis
   as `fluid_v_q` / `fluid_v_r`). The `magnetic_field()`
   accessor returns `(&[Real], &[Real])`; a new
   `magnetic_field_magnitude(cell) -> Real` derives the scalar
   for callers that only want it
   (`sqrt(B_q² + B_r²)` via Q32.32 sqrt).
2. **`Magnetism` law** (`sim/physics/src/magnetism.rs`):
   - `init_field` writes the dipole at planet build time. For
     each cell at axial `(q, r)`:
     ```text
       B_q[i] = 0                                      // axis-aligned dipole
       B_r[i] = -dipole_strength · lat_factor          // toward north pole
       lat_factor = 1 - |r - half_h| / half_h          // 1 at equator, 0 at pole
     ```
   - `integrate` re-derives the field each macro-step with a
     diurnal modulation:
     ```text
       factor = 1 + diurnal_amplitude · triangle(macro_step % period)
       B_r[i] = -dipole_strength · lat_factor · factor
     ```
     `triangle(...)` rises 0→+1 over `period/2` steps, falls
     +1→-1 over the next half (pure integer arithmetic — Q32.32
     has no `sin`/`cos` and the triangle captures the same
     daily-swing pattern at our resolution).

Per-planet `dipole_strength` from `Magnetosphere` class:
- `None` → 0 (dipole disabled, law is a no-op)
- `Weak` → 10
- `Strong` → 50

Default `diurnal_amplitude = 0.05` (±5 %), `diurnal_period = 1`
(once per macro-step ≈ once per sim-day).

**Why this respects the project goals.**
- *Physics as close to real as possible:* magnetism is the only
  vector field on the planet that *every* habitable world has
  (or doesn't, with consequences — atmospheric stripping,
  aurora behaviour, magnetic-sense recognition). Q94 makes it a
  real first-class state. Future Lorentz coupling on charged
  particles, magnetic recognition templates, and aurora
  templates can all read the genuine vector.
- *Least hardcoded values:* the dipole magnitude per
  `Magnetosphere` class is the only hardcoded mapping (3 numbers
  total). `diurnal_amplitude` and `diurnal_period` are tunable
  properties of the law, not per-planet authoring. The
  latitude profile flows from grid geometry.
- *Determinism:* `state.macro_step()` (Q93's clock) is the only
  state-derived input to `integrate`; `init_field` reads only
  `grid.height()`. No interior mutability, no per-tick
  allocation beyond a single `cells` snapshot vector.
- *Reusability:* `state.macro_step()` is the same clock Q93
  tides reads. Adding a second time-varying forcing reused the
  abstraction Q93 introduced rather than re-inventing it.

**Numerical care.**
- `sqrt(B_q² + B_r²)` for the scalar accessor uses the existing
  `sim_arith::transcendental::sqrt`. Magnitudes stay < 100 for
  any realistic dipole strength, so `B²` ≤ 10⁴ — far inside
  Q32.32 range.
- The `diurnal_factor` triangle wave uses `i64::try_from` on
  `u64` phase / half values to keep clippy happy on
  `cast_possible_wrap`; no actual wrap risk for any practical
  `diurnal_period`.
- `integrate` overwrites the field rather than scaling in place
  — same pattern as Q90 radiation, so per-tick error doesn't
  compound across ticks.

**Lives in:** `sim/physics/src/magnetism.rs` (the law),
`sim/physics/src/state.rs` (vector state + magnitude helper),
`sim/physics/src/lib.rs` (re-export),
`sim/physics/src/orchestration.rs` (wired into the macro-step
loop after EM, before chemistry; `integrate_civ_step` takes
`magnetism: Option<&Magnetism>` so deterministic micro-tests
can pass `None`),
`sim/core/src/lib.rs` (`build_laws` returns a 9-tuple including
`Magnetism::for_strength(dipole_strength)` derived from
`Planet::magnetosphere`; `magnetism.init_field(&mut state)` runs
once after state construction).

**Follow-ups (not blocking).**
- *Lorentz coupling on charge.* `F = q·v×B` deflects charged
  particle motion. With Q91 winds (carrying tiny atmospheric
  charge) plus Q94 vector B, a small drift current emerges.
  Wires through once `Wind` exposes a per-cell charge-density
  proxy.
- *Magnetic recognition templates.* Replace the
  `magnetic_field_strong` template's `Field::Charge` proxy with
  a real `Field::MagneticMagnitude` reader. Authoring lands
  alongside a proper `MagneticSense` channel with calibrated
  thresholds.
- *Aurora templates.* Template that fires near the magnetic
  poles (high `|B_r|` × high charge influx). Real aurora
  signature.
- *Reversal events.* On geological timescales (way too slow
  for our 24,000-tick max) the dipole flips polarity. Could
  layer in as a rare-event swing if narrative needs it.
- *Off-axis dipole.* Real planetary dipoles are tilted ~10°
  from the rotational axis. Lands when `Planet` carries a
  magnetic-axis-tilt field separate from its rotational axis.
- *Cosine bulge profile.* Same trig follow-up as Q93: replace
  triangular `lat_factor` with `cos(latitude)` once Q32.32 trig
  arrives.
