## Q62 — Spatial-over-time rendering (post-run keyframes + live ASCII viewport)
**Status:** Decided
**Context:** The project's vision says civilizations rise and fall
over thousands of years. The post-run report already shows two
parts of that arc:

- **Final spatial state** — per-civ territory map at end-of-run.
- **Temporal sparkline** — per-civ `▁▂▃▄▅▆▇█` showing
  `claimed_cells.len()` over time (count only, no spatial info).

What's missing is the *spatial story over time*: where civs grew,
where successors picked up the cells of the fallen, where civs
collided spatially in time. A reader of the report sees that
civ 3 grew to 30 cells at peak then shrank back to 1, but not
*where* on the planet that arc played out.

Separately, the AGENTS.md vision had a "no live UI" rule. The
original framing treated CLI event-streaming as the live mode and
ruled out interactive viewports. That rule was set when the project
didn't have ASCII rendering infrastructure; with the spatial-UI
pass landed (per-cell BFS settlement labels, terrain symbols,
climate strip), a live ASCII viewport is now a small reach and
makes "watch the planet evolve" an option without changing the
project's terminal-only character.

Q62's question: how do we add (a) spatial-over-time keyframes to
the post-run report and (b) a live ASCII viewport CLI option,
without writing two visually-divergent renderers?

**Decision:** Three pieces, single shared renderer:

1. **`frame::render_world_frame(planet_map, planet, frame, caption)`** —
   pure function in `sim_report::frame`. Takes a `WorldFrame` (a
   snapshot of all-active-civs-at-a-tick: per-civ id, claimed
   cells, centroid) and returns one ASCII frame. The only function
   that knows how to draw the world.

2. **Keyframe report** — `Digest::keyframes(every_ticks)` derives
   `WorldFrame`s from the per-civ `territory_history` already
   captured during event replay. The post-run report renders
   ~6 frames at adaptive intervals (5000-year run → 1000-year
   frames; 50000-year run → 8000-year frames, snapped to a clean
   500-year multiple) stacked vertically with year captions.

3. **Live ASCII viewport** — `viewport::ViewportEmitter<W>` is a
   streaming `Emitter` that mirrors the same per-civ claim state
   from arriving events (`Planet`, `PlanetMap`, `CivFounded`,
   `CivTerritoryChanged`, `CivCollapsed`, `Tick`, `RunEnd`) and
   re-renders to stdout every `frame_every` ticks via ANSI
   alternate-screen + cursor-home. Wired to a new
   `--cli=viewport` mode in the `ages` binary, controlled by
   `--frame-every-ticks N`.

Both consumers feed the same `render_world_frame`, so the live
frame and the post-run keyframe are visually identical — there is
no second renderer to drift.

**Per-cell symbol overlay precedence:**

1. Centroid of a civ → uppercase letter (`A` = civ 1, `B` = civ 2,
   …) — founder attention focus / capital.
2. Claimed by exactly one civ → that civ's digit (`1`–`9`, `*` for
   civs ≥ 10).
3. Claimed by multiple civs → `#` (territory dispute).
4. Unclaimed → terrain symbol (▲ peak / △ mountain / ▒ inland /
   ░ coast / `~` shallow / ≈ deep).

The centroid letter overrides the digit on the centroid cell so
the capital reads at a glance even when adjacent cells share the
same civ. Disputed-cell `#` overrides the digit so spatial
collisions between civs surface visibly.

**Why not native GUI.** Project framing remains terminal-only.
The viewport ships in the same Rust crates as the headless run,
no GUI dependency, no second build target. Operator can pipe to
file, attach over SSH, run in tmux, etc. — the existing CLI
ergonomics carry over.

**Why ANSI alternate-screen.** Restores the user's scrollback on
exit (Ctrl-C or natural run-end via `RunEnd`), so the viewport
doesn't pollute the terminal with thousands of lines of refresh
frames. The non-alt-screen path (`use_alt_screen: false`) is for
tests and file-piping consumers.

**Determinism contract holds.** The viewport sits *outside* the
sim — the NDJSON event log is bit-for-bit identical regardless
of whether `--cli=viewport`, `--cli=quiet`, or any other mode is
selected. The viewport is a pure observer of the event stream.
Verified: same seed produces byte-identical NDJSON across
`--cli` settings.

**Why not implement tick=day instead.** The other tractable
deferred item, tick=day, has no consumer that needs sub-year
granularity (the question would be: "what physics fires faster
than a year?"). The viewport would actually be *worse* with
tick=day — a 5000-year run becomes 1.8M frames, an hour of
playback per civ. Year-grained ticks are the right cadence for
both the report keyframes and the live viewport.

**Lives in:** `sim/report/src/frame.rs` (`WorldFrame` +
`render_world_frame`), `sim/report/src/digest.rs`
(`Digest::keyframes`), `sim/report/src/render.rs`
(`render_world_keyframes` section), `sim/report/src/viewport.rs`
(`ViewportEmitter`), `ages/src/main.rs` (`--cli=viewport` +
`--frame-every-ticks` plumbing).

**Follow-ups.**
- Per-frame status line could include population totals or recent
  catastrophes; current line shows tick + active-civ count +
  founded/collapsed totals.
- Adaptive frame-cadence in the live viewport — currently fixed
  via `--frame-every-ticks`, could auto-tune to wall-clock
  budget if a user explicitly asks for "60 fps".
- Color mode (true-color or 256-color) for the viewport. Defer
  until a consumer specifies a colour palette; ASCII-only stays
  the default for terminal-portability.
