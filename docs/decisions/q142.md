## Q142 — Disputed-cell render order
**Status:** Decided
**Context:** User running on Lumen-h (seed 495) reported seeing
`#` glyphs (territorial dispute) but only one capital letter
visible, even with two civs in the run. Investigation:

- Lumen-h triggers Q47 succession at year 44: civ 2 spawns
  with `parent_civ_id = 1` and inherits a subset of civ 1's
  claimed cells. Civ 1 is *not* immediately collapsed —
  both civs are alive simultaneously, claiming overlapping
  territory.
- `viewport.rs::apply_state` derives each civ's `centroid`
  as `claimed_cells.iter().next()` (the smallest cell id in
  the BTreeSet). Both civ 1 and civ 2 have cell 0 as their
  smallest claim → both centroids point at cell 0.
- `frame.rs::render_world_frame_inner` builds a `BTreeMap<cell,
  civ_id>` of centroids by iterating `frame.civs` and inserting:
  ```rust
  for civ in &frame.civs {
      centroids.insert(civ.centroid, civ.civ_id);
      ...
  }
  ```
  `frame.civs` comes from `self.civs.values()` (BTreeMap iteration
  in civ_id order), so civ 1 inserts first, civ 2 inserts
  second and *overwrites* civ 1's entry.
- The render dispatch checked centroids before owners:
  ```rust
  if let Some(&civ_id) = centroids.get(&cell) {
      (centroid_symbol(civ_id), Some(civ_id))
  } else if let Some(claims) = owners.get(&cell) { ... }
  ```
  So at cell 0 the centroid map returned civ 2 → `B` rendered;
  civ 1's `A` was silently masked. The dispute (multi-owner)
  signal never surfaced because the centroid path took
  precedence.

The user saw `#` on the *border* cells (claimed by both, not
centroid for either) but only `B` at cell 0 (claimed by both,
*centroid for both*). The expected "dispute at the contested
capital" wasn't rendering.

**Decision:** Run the dispute check before the centroid lookup.
A cell claimed by 2+ civs always renders as `#`, even if it's
a centroid for one (or both) of them.

```rust
// Q142
let owners_here = owners.get(&cell);
let disputed = owners_here.is_some_and(|c| c.len() > 1);
let (symbol, color_civ) = if disputed {
    ('#', None)
} else if let Some(&civ_id) = centroids.get(&cell) {
    (centroid_symbol(civ_id), Some(civ_id))
} else if let Some(claims) = owners_here {
    (claim_symbol(claims[0]), Some(claims[0]))
} else {
    (terrain_symbol(...), None)
};
```

Result on Lumen-h: row 0 starts with `##` (two contested cells)
instead of `B#`. Both civs lose their letter at the contested
capital — which is honest, the dispute *is* the news.

**Why this respects the project goals.**
- *Truthful rendering.* The previous behaviour silently picked
  one civ as winner of the centroid cell (whichever sorted
  later by civ_id). Now the user sees the real state: contested.
- *No protocol or state change.* Pure presentation reorder in
  `render_world_frame_inner`.
- *Determinism preserved.* The fix is a pure function of
  `owners` and `centroids` maps; same seed → same output.

**Numerical care.**
- `owners.get(&cell)` is called twice in the new layout (once
  for the dispute check, once for the single-claim branch).
  `BTreeMap::get` is O(log n) on the live `civs` count
  (≤ ~20 in practice); negligible.
- `is_some_and` was stabilised in Rust 1.70; project's MSRV
  (`rust-version.workspace = true`) is well past that.

**Lives in:** `sim/report/src/frame.rs::render_world_frame_inner`.

**Follow-ups (not blocking).**
- *Density-frame parity.* `render_density_frame` has the same
  centroid-overlap pattern (line 419) but uses density glyphs
  instead of `#`. Fix would require building a separate
  `owners` map there. Defer until the post-run report's density
  keyframes show the same masking.
- *Ambiguity-aware centroid letter.* When two civs share a
  centroid cell *and* the user has the post-run report open,
  showing the dispute at one capital but the heritor's letter
  at the other (e.g. if civ 2's claim shrinks and civ 1 retains
  cell 0 alone later) requires per-frame disambiguation.
  Current Q142 fix handles only the live overlap moment.
