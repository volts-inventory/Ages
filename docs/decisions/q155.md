## Q155 — Distinct successor centroid + tiny-territory auto-collapse
**Status:** Decided
**Context:** Two civ-lifecycle bugs surfaced together while watching
seed-495 long runs and reading the M4 viewport.

1. **Successor centroid collision.** When civ 2 spawns via Q47
   succession with `parent_civ_id = civ_1`, `Civ::refound_from_stateless`
   sets `territory_centroid = figures[0].cell_assignment`. The
   founding-band code is deterministic on `(species_seed,
   civ_id)`, but the band-id sequence routinely lands the first
   figure on cell 0 — same cell the parent civ used. Q146 fixed
   the *display* (older civ's `A` wins on the contested cell so
   the user still sees a centroid letter there), but the
   underlying field stayed shared. Two civs claiming the same
   capital cell is a data error, not a render policy: the
   simulator was telling the truth that the new capital was
   identical to the old one.

2. **Lingering 1-cell civs.** After a Q47 successor squeezes the
   parent down to a single cell, no existing Q49 trigger fires
   on the parent: food security stays healthy on a 1-cell claim
   with low population, knowledge plateau is on a long timer,
   and cultural lock requires high dogmatism. The parent
   persists indefinitely as a 1-cell holdout, which reads as a
   simulator bug ("why is civ 1 still alive after civ 2 took
   the continent?"). Q49 left "auto-collapse on territory loss"
   as an open sub-question; Q155 closes that specific case.

**Decision (two pieces):**

1. **Successor centroid distinctness.** A new helper
   `sim_civ::pick_successor_centroid(parent_centroid, claimed,
   fallback, grid) -> u32` selects the new civ's `territory_centroid`
   under three rules, in order:

   - **Adjacent-to-parent.** Prefer a `claimed` cell that is one
     of the parent centroid's six hex neighbours. Reads
     narratively as the successor settling on the parent's
     frontier rather than spawning at an arbitrary distance.
   - **Any other claimed cell.** If no adjacent neighbour is in
     `claimed`, return any claimed cell != parent's centroid
     (lowest cell-id wins via `BTreeSet` order).
   - **Fallback.** If `claimed` is empty or every claimed cell
     coincides with `parent_centroid`, return `fallback`.
     Preserves prior behaviour rather than panicking on
     degenerate inputs.

   The helper runs at the two refound sites in `sim/core` —
   stateless-refound (Q47/Q48) and breakaway (Q48-v3). After
   shifting the centroid, the territory is re-BFS'd from the new
   capital so the BFS-anchor and `territory_centroid` agree, and
   the parent's centroid is removed from the successor's claim.
   The drop is what lets the viewport's smallest-claimed-cell
   centroid heuristic resolve the two civs to *visually distinct*
   centroid letters: without it, BFS from a neighbour of cell 0
   still picks up cell 0 (it's adjacent), and civ 1's `A` keeps
   masking civ 2's `B` at the contested cell. The hole gets
   backfilled the moment the successor expands — Q57's BFS
   re-claims the parent's old centroid on a future tick.

2. **Tiny-territory auto-collapse.** A new collapse trigger
   `CollapseReason::TerritoryTooSmall` (tag
   `"territory_too_small"`) fires when a civ holds
   `claimed_cells.len() == TINY_TERRITORY_CELLS` (currently 1)
   for `TINY_TERRITORY_STREAK_TICKS` consecutive ticks
   (currently 24, ~2 sim-years). A new per-civ counter
   `tiny_territory_streak: u64` lives on `Civ`; it increments
   each tick the territory sits at the floor and resets the
   moment it recovers. Triggered alongside the existing food-
   crisis / knowledge-plateau / cultural-lock checks in
   `Civ::check_collapse`.

   The trigger ignores `claimed_cells.is_empty()` (legacy /
   pre-territory state for fresh `Civ::new` callers that haven't
   run `claim_cells` yet). Without that guard, every civ would
   fire `TerritoryTooSmall` before any other trigger could ever
   run, breaking pre-existing tests that exercise food-crisis
   and plateau on bare-bones `Civ::new` instances.

**Why this respects the project goals.**
- *Determinism.* Both rules are pure functions of inputs already
  in scope — the parent's centroid (read once before mutation),
  the successor's BFS-derived `claimed_cells`, the canonical hex
  neighbour order. Same seed → same centroid pick → same claim
  set, byte-for-byte. The streak counter is a per-civ `u64` that
  ticks deterministically with the rest of the lifecycle phase.
- *Q47/Q146 alignment.* The display fix in Q146 stays —
  older-wins on a contested centroid is still the right policy
  in the rare case `pick_successor_centroid` falls back. Q155
  reduces the contention case rather than replacing the display
  rule.
- *Q49 closure.* The decision doc for Q49 already listed
  "auto-collapse on territory loss" as a deferred sub-question.
  Q155 covers the tiny-territory case; conquest collapse and
  plateau-plus-drain remain deferred per Q49's v2 notes.
- *Minimal surface area.* No protocol additions, no event-shape
  changes, no Agent A (sim/report) or Agent B (sim/world) churn.
  The fix lives in `sim/civ` (helper + struct field +
  `check_collapse` line) and the two refound sites in
  `sim/core`.

**Numerical care.** `tiny_territory_streak` is a `u64` with
`saturating_add(1)` per tick, comfortably outside any overflow
window for sim-realistic horizons. `pick_successor_centroid`
uses `u32` cell ids and `BTreeSet` membership tests — no
arithmetic on `Real`/Q32.32, no float operations, no
seed-dependent floating-point ordering. Both pieces sit in the
existing CivLifecycle phase ordering so phase-ordering tests
(Q31) don't shift.

**Lives in:**
- `sim/civ/src/lib.rs` — new `pick_successor_centroid` helper;
  new `Civ::tiny_territory_streak` field initialised to 0 in
  both constructors; new `CollapseReason::TerritoryTooSmall`
  variant; new `TINY_TERRITORY_CELLS` and
  `TINY_TERRITORY_STREAK_TICKS` constants;
  `Civ::check_collapse` updated to maintain the streak and emit
  the new reason; tests for collapse firing, streak reset on
  recovery, helper rule-1 / rule-2 / fallback, and an end-to-end
  Q47-style succession check.
- `sim/core/src/lib.rs` — both refound paths
  (`refound_from_stateless` for Q47-style succession and the
  breakaway path for Q48-v3) read the parent's centroid before
  mutation, override the successor's `territory_centroid` via
  `pick_successor_centroid`, re-run `compute_territory` from the
  new capital, and drop the parent's centroid from the
  successor's claim.
- `docs/decisions/q49.md` — added a "Q155 covers
  territory-shrink collapse" note; Q49 stays open for the other
  deferred triggers.

**Follow-ups (not blocking).**
- *Tune `TINY_TERRITORY_CELLS` upward.* Current value 1 catches
  the literal squeezed-to-one-cell case. A future tuning pass
  may bump this to 2 or 3 to also catch "civ that's been a tiny
  remnant for decades" — but that needs reference-run
  calibration to avoid flushing legitimately small civs (Q49
  notes that tuning is under Q28's umbrella).
- *Q156 territory-claim scoping.* Successor founding currently
  claims any cell type. Q156 will scope claims to land cells and
  scale capacity by terrain — the centroid pick here will stay
  correct under that follow-up because
  `pick_successor_centroid` operates on `claimed_cells` (the
  output of the territory-computation, whatever its rules).
- *Surface the centroid in `CivFounded` directly.* Today the
  viewport derives the centroid from the smallest claimed cell.
  A future protocol bump could add `territory_centroid: u32` to
  `CivFounded` so consumers read the field instead of inferring
  it; the parent-centroid drop in the claim would then become
  unnecessary. Out of scope for Q155 (avoids touching
  `sim/report` and the protocol shape).
