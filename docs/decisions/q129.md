## Q129 — Caption-into-planet, key-above-species, drop tens header
**Status:** Decided
**Context:** Q128 settled the viewport layout at planet → map →
species → key → log with universal centering and axis separators.
Three small follow-ups landed in user feedback:

1. *Section order: legend should sit next to the map.* Q128 put
   the species section between the map and the key, so a viewer
   asking "what's that `▲` glyph in the grid?" had to scroll
   past the species section before reaching the legend.
2. *Caption belongs with the planet.* The caption (`Y{n} M{n} ·
   {civ} civ · {F}F/{C}C`) sat at the top of the map section
   since it came embedded in `frame.rs`'s body output. It reads
   as "this planet, right now" — properly belongs at the bottom
   of the planet section, under the static description.
3. *Tens-digit header row is bloat.* Q122's dense 2-row column
   header (tens digit + ones digit) ensured every column got a
   numerical label, but at 24 columns a viewer can count by 10s
   themselves. The tens row + the blank line above the header
   were two rows of vertical bloat for negligible value.

**Decision:** Three coordinated edits.

1. **Swap species and key sections.** Render order becomes:
   planet → map → key → species → log (was planet → map →
   species → key → log). The legend now sits glance-adjacent to
   the grid glyphs it explains; the species section moves down
   to sit just above the log, where a future per-civ-event
   stream would naturally reference it.

2. **Move the caption into the planet section.** The viewport
   now passes an empty `caption` string to the frame renderer
   so the body comes back without the `Y{n} M{n} ...` line +
   its trailing blank. The caption is then written as the last
   line of the planet section (after the planet card body), so
   the planet section reads top-to-bottom as
   `name → static description → live tick state`. When the
   planet card is hidden via `--no-planet-card`, the caption
   falls back to the top of the map section so the tick state
   never disappears.

3. **Drop the tens-digit header row.** `frame.rs::render_world_frame_inner`'s
   compact branch emits only the ones-digit header (was 2 rows:
   tens digit on top, ones digit below). Columns 10..19 now
   read as `0..9` again on the header — the viewer infers tens
   from position. The horizontal axis separator (Q128 `+----`)
   and per-row `|` are unchanged.

**Sample full layout (seed 1, default flags):**
```
---------------- Lyra-a ----------------
        ocean world · scorching
    methane-rich · 241F · strong mag
      129h · 11mo · 17° · 2 moons
         Y0 M0 · 1 civ · 1F/0C
----------------- map ------------------
         012345678901234567890123
        +------------------------
       0|A11△△▒▒▒░~≈≈≈≈≈≈≈≈≈≈≈≈≈1
       ...
      15|111▲▲▲▲△△△△△△△▒▒▒░~≈≈≈≈≈
----------------- key ------------------
  A=cap · 1=civ · #=war · ~sea · ≈deep
     ▲peak · △hill · ▒land · ░coast
-------------- Cyranites ---------------
      centralized medium cognition
    sense: tactile · manip: tentacle
    44y · solitary · noisy · carbon
----------------- log ------------------
      y0 civ 1 founded (10 cells)
```

**Why this respects the project goals.**
- *Reading order matches information flow.* Planet description
  (static) → planet state (live tick) → map (visualisation) →
  key (legend for the visual) → species (who lives there) →
  log (running events). Each section feeds the next.
- *Vertical density.* Q129 saves 2 rows compared to Q128 — the
  tens header + the blank line above the headers — without
  losing any information. On a portrait phone where every row
  matters, that's two more rows of grid or log visible.
- *Determinism preserved.* `frame.rs` change is purely the
  removal of one `writeln!`; viewport reordering is purely
  shuffling the order of `write_centered_*` calls. No new
  state, no per-tick branching.
- *Backward-compatible body API.* Frame functions still accept
  a `caption: &str`; the post-run report (digest path) still
  passes a non-empty string and gets the embedded caption.
  Only the viewport caller passes `""`.

**Numerical care.**
- Caption fallback: when `show_planet_card = false` or
  `Planet` event hasn't arrived, the caption is written at the
  top of the map section instead of vanishing. The flag
  `planet_section_shown` gates this branch.
- Compact header without tens row: `chars().count()` of the
  ones-only header is `row_prefix_pad + grid_width` =
  `3 + 24` = 27 chars in the default 24-wide grid, same width
  as the axis separator and grid rows — alignment preserved.

**Lives in:** `sim/report/src/viewport.rs`
(`render()` passes `""` for caption, writes caption at end of
planet card or top of map; species + key sections swapped).
`sim/report/src/frame.rs` (`render_world_frame_inner` compact
branch drops the tens-digit header row).

**Follow-ups (not blocking).**
- *Optional tens header.* A future flag could re-enable the
  tens row for grids wider than ~30 cols where counting by 10s
  becomes harder. Pure quality-of-life; deferred.
- *Caption placement when no planet section.* Currently falls
  back to the top of the map; could also fall back to its own
  pre-map line ("status: Y0 M0 ..."). Marginal; deferred.
- *Section ordering as a config.* `--section-order
  planet,map,key,species,log` would let users pin their
  preferred reading flow. Probably overkill.
